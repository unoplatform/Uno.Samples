## 1. Product Overview

### Summary

NEXUS is a supervisory control and data acquisition (SCADA) dashboard designed to provide real-time monitoring, control, and analysis of industrial manufacturing operations. It consolidates production line status, equipment health, workforce management, and network infrastructure into a single unified interface for plant operators and supervisors.

### Primary Purpose

Enable manufacturing personnel to monitor factory operations in real-time, track production metrics, manage maintenance schedules, and respond quickly to alerts or anomalies.

### Main Value Proposition

Single-pane-of-glass visibility into all critical manufacturing operations with minimal cognitive load, combining live operational data with historical analytics.

### Platform

Web application optimized for large desktop displays (control room monitors, engineering workstations). The information density and small typography suggest non-mobile usage. Fixed layout implies desktop-first design.

### Core Use Cases

2. **Shift Handoff** — Review current shift progress, batch queue, and pending work before/after shift change

3. **Maintenance Planning** — Check equipment health scores, schedule preventive maintenance, manage spare parts inventory

4. **Performance Analysis** — Analyze weekly output trends, efficiency metrics, and defect rates to identify improvement opportunities

5. **Alert Triage** — Review system log alerts, identify patterns, and take corrective action

### Key Entities & Relationships

| Entity | Attributes | Relationships |
|--------|------------|---------------|
| **Production Line** | ID, name, status, output %, temp, pressure | Contains batches, has maintenance schedule |
| **Batch** | ID, product name, quantity, priority, ETA | Assigned to production line |
| **Work Order** | ID, equipment, type, date, technician, status | References equipment, assigned to technician |
| **Equipment** | Name, health %, next service, runtime hours | Has work orders, consumes spare parts |
| **Spare Part** | Name, in-stock qty, min-stock, status | Used by equipment |
| **User** | Name, email, role, last access, status | Has permissions, assigned to work orders |
| **Alert/Log Entry** | Timestamp, type, message | Generated by system/equipment |
| **Network Node** | ID, label, connections | Part of network topology |

### Hierarchy & Grouping

- **Overview** groups operational KPIs (throughput, efficiency, uptime, energy) separately from tactical data (line status, alerts)
- **Production** separates planning (batch queue) from execution (line status) from resources (materials, shift)
- **Analytics** separates leading indicators (OEE, FPY) from lagging analysis (defects, comparisons)
- **Maintenance** separates scheduled work from inventory from predictive health
- **Settings** separates user preferences from system configuration from user administration

## 4. UX Flows & Behaviors

### Primary User Flows

#### Flow 1: Monitor Production Status

1. User lands on **Overview**
2. Scans 4 KPI metric cards for anomalies
3. Reviews production lines table for line-level status
4. Checks system log for recent alerts
5. *If issue detected:* Navigates to **Production** for detail

#### Flow 2: Manage Batch Queue

1. Navigate to **Production**
2. Review batch queue table (ID, product, qty, priority, ETA)
3. Click "+ NEW BATCH" to add batch *(assumed modal/form)*
4. Monitor shift progress bar
5. Check material inventory for stock levels

#### Flow 3: Schedule Maintenance

1. Navigate to **Maintenance**
2. Review equipment health cards (circular progress indicators)
3. Identify equipment with low health % or upcoming service
4. Click "+ WORK ORDER" to create maintenance task
5. Assign technician and schedule date

#### Flow 4: Analyze Performance Trends

1. Navigate to **Analytics**
2. Review 4 performance metric cards (OEE, MTBF, MTTR, FPY)
3. Examine weekly output bar chart for patterns
4. Review efficiency trend line chart (12-month)
5. Check defect analysis for top issues
6. Use comparison matrix for week-over-week delta

#### Flow 5: Configure System Settings

1. Navigate to **Settings**
2. Toggle system preferences (auto backup, alerts, dark mode)
3. Adjust data retention and refresh rate via dropdowns
4. Review network configuration status
5. Modify alert thresholds via inline inputs
6. Manage users (add, edit, remove)

- **Social features** — No comments, mentions, or collaboration
- **Rich media** — No images, videos, or file attachments
- **Personalization** — Same dashboard for all users (role-based access assumed)
- **Offline support** — Real-time sync indicators suggest always-connected
- **Accessibility (full)** — Small fonts, low contrast text, no visible ARIA hints *(assumption)*
- **Theming** — Dark mode only, no customization beyond settings

## 10. Sample Implementation Specification

> *The following answers address the critical open questions required to build a functional sample/demo.*

### Data Source & Structure

**Decision:** Mock REST API with JSON endpoints. Data is pre-seeded with 30 days of historical records. No actual PLC/SCADA integration — all data is simulated with realistic variance patterns.

**Endpoints:**

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/metrics` | GET | Current KPI values (throughput, efficiency, uptime, energy) |
| `/api/v1/lines` | GET | Production line status array |
| `/api/v1/lines/:id` | GET | Single line detail |
| `/api/v1/batches` | GET, POST | Batch queue management |
| `/api/v1/equipment` | GET | Equipment list with health scores |
| `/api/v1/work-orders` | GET, POST | Maintenance work orders |
| `/api/v1/alerts` | GET | System log entries |
| `/api/v1/users` | GET, POST, PUT, DELETE | User management |
| `/api/v1/settings` | GET, PUT | System preferences |

### Update Frequency

**Decision:** Hybrid polling + WebSocket approach.

| Data Type | Update Method | Interval |
|-----------|---------------|----------|
| Production metrics | Polling | 5 seconds |
| Line status | Polling | 5 seconds |
| Equipment health | Polling | 30 seconds |
| System log/alerts | WebSocket push | Real-time |
| Analytics/charts | On-demand | Tab navigation |
| Settings | On-demand | User action |

### Authentication Model

**Decision:** Simple JWT authentication with role-based access control.

| Role | Dashboard | Production | Analytics | Maintenance | Settings |
|------|-----------|------------|-----------|-------------|----------|
| `admin` | ✓ Full | ✓ Full | ✓ Full | ✓ Full | ✓ Full |
| `supervisor` | ✓ Full | ✓ Full | ✓ Full | ✓ Full | ✓ Partial (no user mgmt) |
| `operator` | ✓ Read | ✓ Read | ✓ Read | ✓ Read | ✗ Hidden |

**Demo Accounts:**

```
admin@nexus.io      / admin123     → Full access
supervisor@nexus.io / super123     → No user management
operator@nexus.io   / oper123      → Read-only, no Settings tab
```

**Token Specification:**
- JWT with 24-hour expiry
- Refresh token rotation enabled
- Stored in httpOnly cookie (production) or localStorage (demo)

### Historical Data Depth

**Decision:** Tiered retention based on data type.

| Data Category | Retention | Granularity |
|---------------|-----------|-------------|
| Production metrics | 90 days | Per-minute readings |
| Equipment health | 90 days | Hourly snapshots |
| Defect records | 30 days | Individual records |
| Analytics aggregates | 12 months | Daily/weekly/monthly rollups |
| System logs | 30 days | Individual entries |
| Work orders | Indefinite | Full records |

## Summary

NEXUS is a purpose-built industrial monitoring dashboard optimized for control room environments. Its high-density, dark-themed interface prioritizes real-time operational awareness over visual polish. The design system is intentionally austere — monospace typography, thin borders, and node-based visual language reinforce the technical/industrial brand positioning.

### Implementation Recommendations

1. **Define the real-time data architecture** (streaming vs. polling)
2. **Design modal flows** for creation actions (batches, work orders, users)
3. **Establish accessibility standards** (font size minimums, contrast ratios)
4. **Build responsive breakpoints** if tablet support is required
5. **Document the alert/notification system** end-to-end
